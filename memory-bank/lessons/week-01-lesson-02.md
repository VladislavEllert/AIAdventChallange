# Неделя 1 — День 2: формат ответа (конспект)

## Задание

Один и тот же запрос с разным уровнем контроля ответа через API:
- явное описание формата;
- ограничение длины;
- условие завершения (stop sequence / инструкция).
Сравнить: без ограничений vs с ограничениями.

Реализация: `week-01/day-02/compare.py` — консольный скрипт, шлёт вопрос
трижды одной моделью (`openai/gpt-4o-mini`), печатает 3 варианта + токены.

## Три варианта в скрипте

- **A** — без ограничений. Длинный вольный ответ (~150–500 токенов выхода).
- **B** — system-промпт (формат: 3 пункта по ≤10 слов) + `max_tokens=150` + `stop=["[КОНЕЦ]"]`. Короткий осмысленный ответ (~40 токенов).
- **C** — `max_tokens=20` без формата. Обрыв на полуслове (ровно 20 токенов).

## Что понял про инструменты контроля

| Инструмент | Модель знает? | Поведение |
|---|---|---|
| system-промпт («3 пункта», «кратко») | да | укладывается осмысленно |
| `stop="[КОНЕЦ]"` | видит маркер в инструкции | умный обрыв на маркере; саму строку API не включает |
| `max_tokens` | **нет** | гильотина: рубит на полуслове по счётчику |

## Ключевые наблюдения

- `stop` режет генерацию в момент появления стоп-строки и **не включает** её в ответ → маркера `[КОНЕЦ]` в выводе не видно. Это и есть доказательство, что stop сработал.
- В B `max_tokens=150` почти никогда не достигается — промпт держит ответ короче. max_tokens здесь = страховка от перерасхода, а не основной ограничитель.
- C иногда _случайно_ заканчивается целым предложением (20-й токен лёг у точки) — это везение счётчика, не ум модели.

## Вывод

Контроль ответа = комбинация: промпт формирует осмысленно, `max_tokens`
страхует бюджет, `stop` гарантирует границу. Так настраивают агентов под
себя — поведение промптом, бюджет и границы параметрами. Контроль = качество
+ экономия токенов.
