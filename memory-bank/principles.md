## Урок 1 — основы LLM и промптинга

**Главный посыл:** неделя про устройство и настройку нейросети изнутри. База.
Ключевой навык — **подбирать модель и параметры под задачу**.

**Акценты:**
- Преодолеть **барьер недоверия** к ИИ (выше грейд — сильнее мешает).
- ИИ как **продвинутый поиск**, но вывод — своей головой. Не давать модели думать за тебя.
- **Новее / больше ≠ лучше.** Решают обучение + архитектура + соответствие задаче.

**Правило параметров:** точная задача (математика) → temperature/разброс вниз; творчество → вверх. Max tokens больше ≠ лучше, просто дороже.

Полный конспект (термины, 6 стадий агента, техники промптинга, параметры LLM, типы промтов, 5 отличий моделей): [`lessons/week-01-lesson-01.md`](lessons/week-01-lesson-01.md).

## День 2 — контроль формата ответа

**Вывод:** ответ модели контролируют три инструмента с разными ролями.

| Инструмент | Роль |
|---|---|
| **system-промпт** | форма + длина **осмысленно** — модель в курсе, сама укладывается |
| **max_tokens** | потолок-бюджет, страховка от перерасхода — тупой, но надёжный |
| **stop** | детерминированная точка обрыва — гарантия границы |

- Сила — в **комбинации**: промпт формирует, max_tokens страхует бюджет, stop гарантирует конец.
- `max_tokens` модель **не знает** — лимит API, рубит на полуслове (гильотина). Хочешь, чтоб модель _старалась_ быть краткой → пиши в промпте.
- **Практический смысл:** агента настраиваешь под себя — промптом задаёшь поведение и формат, параметрами держишь бюджет и границы. Контроль = качество + экономия токенов.
- Пример живьём: **caveman** — system-промпт-контроллер поведения («режь воду»), осмысленный «умный стоп», не гильотина.

Фундамент под неделю 2 (агенты).

Конспект задания: [`lessons/week-01-lesson-02.md`](lessons/week-01-lesson-02.md).

## День 3 — способы рассуждения

**Вывод:** способ рассуждения меняет ответ сильнее, чем подкрутка параметров.
Проверено вживую: 1 вопрос-ловушка, 4 техники, одна модель, temperature=0.

- **Мультиагентность точнее всего.** Группа экспертов (аналитик/инженер/критик)
  дала верный ответ там, где прямой/пошаговый/meta провалились. Агенты круче,
  когда **видят результат друг друга и спорят**: один решает, другой критикует,
  третий синтезирует. Сильнее одиночного CoT-монолога.
- **CoT не спасает при ложной посылке** — аккуратно обоснует неправильный ответ.
- **Промпт решает: намёки не работают, работает жёсткий factual-constraint.**
  Слабая модель цепляется за поверхностный якорь, пока ограничение не **переставит
  постановку задачи**.
- **Температура = точность ↔ разнообразие, НЕ тупее ↔ умнее.** temperature=1 —
  шум: ума не добавил, сломал даже рабочий способ. Точность → temp вниз +
  структура; брейншторм → temp вверх.
- **Self-consistency** — приём в коде (не параметр API): гонять способ N раз при
  temp>0, брать ответ большинством. Гасит шум, не теряя рандом.

Мостик в неделю 2 (агенты): руками доказали — **совместная работа ролей надёжнее
одиночного прохода**.

Конспект задания: [`lessons/week-01-lesson-03.md`](lessons/week-01-lesson-03.md).

## Неделя 2 — агенты (главный вывод)

**Вывод (Влада):** агент — НЕ просто сущность с системным промптом и вызовом
API. Система, где **правильно организованы**: управление контекстом, память,
планирование, вызов инструментов, стоимость токенов, разбиение задач на стадии.
Цель — результат за **малый токен** с **большей точностью**. К этому стремимся.

**Честное состояние на день-6 (старт недели):** построен только **каркас** —
агент-сущность + инкапсуляция вызова + хранение истории (SwiftData,
[`AgentChat/`](../AgentChat/)). «Умного» управления пока НЕТ:
- контекст шлём **наивно** — вся история целиком, **без сжатия/обрезки/окна**
  (НЕ «сжимаем» — шлём as-is, токены растут линейно);
- память только сессионная+persisted на чат; долгой (факты между чатами),
  summary, векторов — нет;
- планирования, настоящих инструментов (tool-calling), подсчёта стоимости,
  стадийности — нет.

**Добить неделю 2:** бюджет контекста (обрезка по токенам), сжатие/summary
старого, долгая память (факты о юзере между чатами), подсчёт токенов/₽ за
сообщение. Инструменты — неделя 4 (MCP).

Полная сводка теории недели: [`lessons/week-02-summary.md`](lessons/week-02-summary.md).

## Неделя 2 — день 8: токены и контекстное окно

**Вывод (Влада):** токены = деньги. Каждый ход вся история пере-отправляется в
модель → вход (prompt) и стоимость растут по ходу диалога. Точные числа даёт API
(`usage`), локальная оценка — грубая.

Модель **сама контекстом не управляет**: при переполнении окна голая LLM просто
падает (400). Управляет **приложение** — окно последних N + сжатие старого
в `summary` (как авто-compact в Claude Code) + долгая память (факты, отдельный слой,
переполнением чата НЕ ломается). Сжатие **лоссовое и избирательное**: держит то, что
счёл важным, остальное теряет → отсюда забывание и галлюцинации. На дешёвых моделях
выжимка кривее → деталей теряется больше.

Реализовано в [`AgentChat/`](../AgentChat/): подсчёт токенов (`usage` + оценка),
стриминг (SSE), компактация в summary, демо-лимит/окно для наглядной поломки.

Урок (день-8): [`week-02/day-08/README.md`](../week-02/day-08/README.md).

## Неделя 3 — stateful-агент + разбор чата (дни 11–14)

**Тема:** профиль + state machine + инварианты = полный stateful-агент.
Конспект: [`lessons/03-memory-state-summary.md`](lessons/03-memory-state-summary.md),
день-11: [`lessons/week-03-lesson-11.md`](lessons/week-03-lesson-11.md).

**Главные мысли из чата (Гладков = авторитет):**
- **Агент = аналог КЛАССА в ЯП** под конкретную задачу, много инстансов, НЕ обязательно отдельный
  файл. Общие черты: входные/выходные условия, system prompt, тулы (вокруг этого Koog). Объект,
  управляющий контекстом, — уже агент. «Не бойся спавнить инстансы агентов.»
- **Агент vs обвязка:** агент = тачка под цель; память/профили/state = асфальт+правила+пит-стопы.
  Плохая обвязка → хреново даже с топ-моделью (тест: включил топ-модель, резко лучше → обвязка плохая).
- **День 12 профиль:** профиль ПОЛЬЗОВАТЕЛЯ (не ассистента). Микс: агент сам заполняет (как memory
  в Claude) + юзер сам редактирует (.md как agents.md). «Стойкость» привычек: часто Python → дальше
  код на Python без явного указания.
- **День 13 FSM:** задача = один user prompt → пайплайн, у каждой стадии свой system prompt.
  Переходы детерминированные (артефакт → дальше; проще+надёжнее) ИЛИ оркестратор. Рекомендация:
  4 агента, по 1 на статус. Пауза = явное прерывание, при закрытии сохранить стейт. Переходы — КОДОМ
  (баг: validation→planning не был в transitions → тупик). Опц. надзорный агент сверху.
- **День 14 инварианты:** подкапотные ограничения (юзер может не знать), проверка код + LLM
  (комбинировать). Звёздочка: юзер сам ставит самозапреты. Взаимопроверка LLM (3 спорящих).
- **Процесс (Denis Suprun):** CLAUDE.md память → сессия на задачу → summary+задание+чат → plan mode →
  правка плана → запуск → сценарии видео. Декомпозиция на атомарные подзадачи стабилизирует.

**Вердикт по платформе (день 12+):** пивот iOS → CLI/TUI на Python. iOS sandbox блокирует недели
4–7 (MCP/RAG/VPS/пайплайны). AgentChat остаётся, новая папка с нуля. Детали — в `CLAUDE.md` (раздел
«Платформа заданий») и плане `~/.claude/plans/agile-floating-bentley.md`.

**Доп. пивот (день 15+):** CLI → Web. Появился `agent-web/` (FastAPI + React). Все фичи CLI
портированы туда. MCP и последующие недели реализуются в web-app контексте.

## Неделя 4 — MCP (дни 16–18)

**Тема:** Model Context Protocol — стандарт взаимодействия AI host с внешними инструментами.

**Главное:**
- MCP = стандарт, не framework. SDK на Python/Kotlin/TS — реализации, не сам стандарт.
- Архитектура: Host+Agent → MCP Client → MCP Server → API/CLI/DB.
- Transport: stdio (локальный subprocess) или HTTP/SSE (remote).
- Tool = имя + description + input schema + handler. Схема → модель выбирает + формирует аргументы.
- **Token overhead:** schemas грузятся в prompt при КАЖДОМ sampling call (idle overhead). 6 tools × 900 ≈ 5400 токенов даже если tools не нужны.
- **MCP vs Skill+CLI:** MCP лучше для remote/корпоративных/multi-host. Skill+CLI лучше для локального — нет idle overhead, загружается on demand.
- MCP не deprecated — для удалённого и корпоративного будет жить долго.
- Безопасность: least privilege, строгая validation, никакой shell interpolation, read-only default.

Конспект: [`lessons/week-04-mcp.md`](lessons/week-04-mcp.md).

**Практика (дни 16–18):**
- MCP-сервер на VPS: FastMCP + 3 tools (web_search, get_moex_quote, get_moex_summary).
  - `host="0.0.0.0"` в FastMCP обязательно — иначе DNS rebinding protection блокирует внешние IP (421).
  - APScheduler кэш MOEX каждые 30 сек в фоне.
  - Деплой: systemd сервис, uvicorn + httptools.
- agent-web tool-calling loop: non-streaming call с `tools=schemas` → если tool_calls → execute → stream final.
  - SSE события tool_start/tool_done → frontend плашка с анимацией пока tool работает.
  - Graceful degradation: если MCP недоступен или модель не поддерживает tools → обычный ответ.

## Неделя 7 — растущий dev-ассистент (дни 31–35)

**Тема:** один и тот же чат-агент обрастает возможностями по дням — вторая RAG-база
по себе самому (день 31), AI-ревью PR (день 32), саппорт по тикетам (день 33),
файловый агент с human-in-the-loop (день 34), автоматизация рутины курса (день 35).
Не 5 разных приложений — один `agent-web`, фичи по модулям.

**Главное:**
- **Danger-gate без LLM.** День 34: `danger_level(tool_name) -> safe|dangerous` — чистая
  функция, lookup table, без единого вызова модели. LLM в пути авторизации даёт
  недетерминированное решение о безопасности плюс латентность на каждый вызов — для
  бинарного классификатора "эта операция пишет на диск или нет" это строго хуже
  регулярного словаря. Тот же принцип день 35 повторяет для верификатора ритуала
  (формат таблицы/наличие ссылки — механическая проверка, не творческая задача).
- **Human-in-the-loop — это реальный modal + реальный эффект на диске, не иллюзия
  подтверждения.** День 34: опасную операцию нельзя одобрить с мёртвой вкладки —
  confirm-стор различает "сессию" (весь SSE-запрos) и "call_id" (конкретный тул-вызов),
  протухшая сессия автоматически отклоняет поздний approve. Живой баг, пойманный только
  браузером (не pytest): JSON-дамп аргументов без ограничения высоты — `write_file` с
  большим `content` толкает кнопки Allow/Deny за пределы экрана, недостижимо даже для
  живого человека, не только для Playwright.
- **Именование тулов — общий процесс-широкий singleton, коллизии реальны.** День 35:
  MCP-сервер (день 31) и новый локальный `services/tools/registry` — два независимых
  источника тулов, мёржатся в один список для LLM. Тул с тем же именем в обоих —
  не косметика, а потенциальный `400` от OpenAI API (дубли имён функций запрещены) на
  КАЖДЫЙ чат с тулами, не только в новой фиче. Решение: не регистрировать
  безопасные read-only дубликаты в общий реестр, если MCP уже отдаёт то же имя.
- **`git diff` репозитория, который не коммитился неделю, — не то же самое, что "diff
  за сегодня".** День 35: наивный `diff[:N]` срез текста, отсортированного git
  алфавитно по путям, систематически теряет только что созданные файлы (они физически
  не попадают в окно среза). Приоритизация untracked/added файлов перед modified —
  единственный надёжный сигнал "это работа именно сегодняшней сессии" в репозитории без
  дисциплины ежедневных коммитов.
- **Верификатор без источника правды по определению отклоняет любое утверждение.**
  День 35: пайплайн ритуала физически не имеет способа узнать, была ли запись видео —
  значит ЛЮБОЕ значение колонки "Видео" кроме буквального `todo` недоказуемо по
  построению и отклоняется, а не проверяется эвристикой на "похоже на реальную ссылку".
- **Сравнение промптов — минимум одна ось контроля.** День 35: `v1` vs `v2_strict`
  (явный чеклист реальных классов багов проекта вместо общей просьбы "найди баги")
  сравнивались в ОДНУ сессию на ТОМ ЖЕ RAG-индексе — recall 33%→100% на 4 фикстурах
  изолированно приписан промпту, а не шуму (день 32→33 сравнение той же метрики без
  этой дисциплины спутало пересборку индекса с обычным разбросом LLM).
