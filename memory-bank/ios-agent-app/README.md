# iOS Agent App — спецификация

Личное iOS-приложение: мульти-агентный чат поверх ProxyAPI (OpenAI-совместимый).
Цель — первое приложение под собственный iPhone 13, песочница для всего, что
прошли в Неделе 1 (модели, параметры, промпты), + база под Неделю 2 (агенты).

Статус: **спека готова, реализация — на выходных (старт ~июнь 2026, суббота).**
Подробные милстоуны — см. `milestones.md`.

---

## 1. Что строим (одной фразой)

Приложение, где есть несколько ИИ-агентов (готовые + созданные пользователем),
у каждого свой системный промпт, своя модель и свои параметры генерации, своя
память. Внутри агента — несколько чатов. Красивый дизайн, светлая/тёмная тема.

## 2. Зафиксированные решения

| Решение | Выбор | Почему |
|---|---|---|
| UI-фреймворк | **SwiftUI** | декларативно, dark mode из коробки, быстрее красивый UI |
| Deploy target | **iOS 17+** | `@Observable` + SwiftData, чистый код; iPhone 13 тянет iOS 18 |
| State | **@Observable** (Observation) | меньше boilerplate чем ObservableObject |
| Persistence | **SwiftData** | нативно для iOS 17, хранит агентов/чаты/память/настройки |
| Сеть | **URLSession + Codable** | ProxyAPI OpenAI-совместим, SDK не нужен |
| Архитектура | **MVVM** | View → ViewModel (@Observable) → Service → ProxyAPI |
| Ответы LLM | **SSE-стриминг** | печать по токенам как в ChatGPT |
| Дизайн | SwiftUI итеративно | пишем код сразу, правим в превью |

## 3. Архитектура (слои)

```
View (SwiftUI)
  → ViewModel (@Observable)        состояние экрана, экшены
    → ChatService                  оркестрация: собрать контекст, вызвать API, стримить
      → ProxyAPIClient             URLSession, SSE-парсинг, Codable DTO
    → MemoryStore                  чтение/запись памяти агента и чата
  ↕ SwiftData (ModelContainer)     персистентность всех сущностей
```

Принцип: View тупой, вся логика в ViewModel/Service. ProxyAPIClient ничего не
знает про агентов — только «дай messages + params → верни поток токенов».

## 4. Модель данных (SwiftData @Model)

```
Agent
  id: UUID
  name: String
  emoji: String                 // иконка-эмодзи (просто, без ассетов)
  systemPrompt: String          // личность агента
  model: String                 // id из массива моделей в коде (см. §7)
  params: GenerationParams      // встроенный тип (см. ниже)
  memory: AgentMemory           // долгая память агента (между чатами)
  chats: [Chat]                 // relationship, cascade delete
  createdAt: Date
  isBuiltIn: Bool               // пресет (Бро/Поисковик) нельзя удалить

Chat
  id: UUID
  title: String                 // авто из первого сообщения или вручную
  messages: [Message]           // история ЭТОГО чата (короткая память)
  summary: String?              // сжатая выжимка чата → питает AgentMemory
  createdAt: Date

Message
  id: UUID
  role: Role                    // .system / .user / .assistant
  content: String
  createdAt: Date

GenerationParams (struct, Codable)
  temperature: Double
  topP: Double
  topK: Int?                    // не у всех моделей (см. §7)
  seed: Int?
  maxTokens: Int

AgentMemory
  facts: [String]               // накопленные факты/предпочтения о юзере
  updatedAt: Date
```

## 5. Память — двухуровневая (ключевая фича)

Это то, что отличает app от голого чата. Два уровня:

**Уровень 1 — память чата (короткая).**
Внутри одного чата модель видит всю историю `Chat.messages`. Заходишь в агента →
создаёшь чат → внутри него идёт обычный диалог с полным контекстом.
Когда история длинная — обрезаем по бюджету токенов (последние N сообщений +
system-промпт). Это контекстное окно чата.

**Уровень 2 — общий memory bank агента (долгая, между чатами).**
У агента есть `AgentMemory.facts` — копилка фактов о пользователе/договорённостей,
которая переживает закрытие чата и видна во ВСЕХ чатах этого агента.

Как наполняется:
- При закрытии/сворачивании чата (или по кнопке «запомнить») — отдельный
  дешёвый вызов LLM: «выдели из диалога стойкие факты о пользователе → список».
  Результат сливаем в `AgentMemory.facts` (дедуп).
- Альтернатива на старте (проще): кнопка «Запомнить выделенное» — юзер сам
  кидает факт в банк. MVP без авто-извлечения.

Как используется:
- При старте любого чата агента факты из `AgentMemory` инжектятся в system-контекст:
  `system = agent.systemPrompt + "\n\nЧто ты помнишь о пользователе:\n- факт1\n- факт2"`.
- Так «Бро» помнит, что ты любишь, даже в новом чате.

⚠️ Тонкость: банк растёт → токены растут. Нужен потолок (топ-N фактов или
периодическое сжатие банка отдельным вызовом). На MVP — просто лимит на длину.

## 6. Агенты

**Пресеты (isBuiltIn, нельзя удалить, можно настраивать):**
- **Бро** 🤝 — друг, общается по-человечески на любые темы. systemPrompt =
  неформальный, эмпатичный, без морализаторства.
- **Поисковик** 🔎 — отвечает по делу, как Google/энциклопедия, факты + кратко.
  systemPrompt = точность, структура, низкая temperature по умолчанию.

**Пользовательские:** юзер создаёт агента — задаёт имя, эмодзи, системный промпт,
выбирает модель и параметры. Полная свобода.

## 7. Модели и параметры (переключение в коде)

Модели держим массивом в коде (через ProxyAPI роутинг по имени). Пример:

```swift
enum LLMModel: String, CaseIterable {
    case geminiFlashLite = "gemini/gemini-2.5-flash-lite"
    case geminiFlash     = "gemini/gemini-2.5-flash"
    // добавлять рабочие id, сверяя живым вызовом models.list() (см. CLAUDE.md)
}
```

⚠️ ВАЖНО (квирки, проверять под каждую модель):
- Gemini id в ProxyAPI **с префиксом `gemini/`**, без него → 400.
- `gemini-3.1-flash-lite` НЕ роутится — не использовать.
- Доступность параметров зависит от провайдера:
  - OpenAI: есть `seed`, **нет** `top_k`.
  - DeepSeek: `temperature` диапазон [0;2], **нельзя** `temperature` и `top_p`
    вместе, **нет** `top_k`/`seed`.
- Рабочие id и допустимые параметры сверять **живым запросом**, не доками.

UI-стратегия параметров: на MVP показываем все крутилки, ловим 400 от API и
показываем человеку («модель не поддерживает top_k»). Позже — прятать
недоступные под выбранную модель. Это сознательный learning-кейс из Недели 1.

Крутилки per-agent (в настройках агента): `temperature`, `top_p`, `top_k`,
`seed`, `max_tokens`. Меняешь → сравниваешь поведение. Прямое продолжение
day-03/day-04 (игра с temperature), но в живом app.

## 8. Экраны (SwiftUI)

1. **Список агентов** — карточки (эмодзи, имя, модель), кнопка «+ создать».
2. **Агент → список чатов** — чаты агента, кнопка «+ новый чат»,
   доступ к настройкам агента и его memory bank.
3. **Чат** — лента сообщений, стриминговая печать ответа, поле ввода.
4. **Настройки агента** — имя, эмодзи, системный промпт, выбор модели,
   крутилки параметров, просмотр/правка memory bank.
5. **Глобальные настройки** — переключатель темы (светлая/тёмная/системная),
   ввод/хранение API-ключа (см. §9).

Дизайн-планка: «как будто делал проф-дизайнер» — семантические цвета,
SF Symbols, аккуратная типографика, отступы по сетке, плавные анимации,
полноценная тёмная тема через семантику (не хардкод цветов).

## 9. Безопасность API-ключа 🔴

Главный риск нативного app: **ключ ProxyAPI попадает в бинарь и извлекаем**.

Текущий этап (личное использование на своём iPhone):
- Ключ **НЕ хардкодить в исходники, НЕ коммитить**.
- Хранить через `Secrets.xcconfig` (в `.gitignore`) → пробрасывать в Info.plist,
  либо вводить ключ руками в настройках app и класть в **Keychain**.
- Keychain предпочтительнее: ключ не в бинаре вообще, юзер вводит один раз.

Этап Недели 6 (будет VPS):
- Поднять **бэкенд-прокси** на серваке: app стучит к своему серверу, сервер
  держит ключ ProxyAPI и ходит в него. Ключ с устройства уходит совсем.
- Тогда же app можно безопасно «поставить на сервак и пользоваться».

Правило: пока бэкенда нет — app только для себя, ключ в Keychain, в стор/чужим
не отдавать.
