# Summary: неделя 2. Инструментарий, контекст, память и агенты

## Главная тема недели

Неделя 2 про внутреннее устройство LLM и AI-агентов: как работают с контекстом, памятью, планированием, инструментами.

Главная мысль: чтобы делегировать агентам больше, понимай как обрабатывают контекст, «помнят» данные, строят план, вызывают инструменты.

## Из чего состоит агент

Четыре блока:

1. **Context management** — управление контекстом.
   - Агент работает только с тем, что в контексте или подгружено.
   - Качество контекста = качество ответа.

2. **Memory management** — управление памятью.
   - Память не встроена в модель.
   - Организуй отдельно: сохраняй, сжимай, векторизуй, доставай, подмешивай в контекст.

3. **Planning** — планирование.
   - Агент определяет цель, бьёт на задачи, выбирает стратегию, выполняет.

4. **Tool orchestration** — управление инструментами.
   - Агент выбирает инструмент, вызывает, получает результат, интерпретирует, обрабатывает ошибки, обновляет состояние.

## Контекст-менеджмент

### У LLM нет настоящей памяти

Модель не помнит юзера. Каждая сессия стартует без памяти, если данные не сохранены и не подгружены заново.

Ощущение «помнит» создают внешние механизмы: сохранённые файлы, профили, память, векторные базы, системные подстановки. Эти данные идут в новую сессию как часть контекста.

### Контекст влияет на следующие ответы

Внутри сессии каждый промпт влияет на дальнейшее поведение. Уточнения, оценки, направление разговора перебалансируют влияние прошлых сообщений.

Системный промпт — самый важный уровень инструкций, но длинный противоречивый контекст ухудшает поведение агента.

### Роли и инструкции критически важны

LLM обучены на огромных данных, поэтому без ограничений модель угадывает нужную область. Задавай явно:

- роль;
- задачу;
- стек;
- формат результата;
- ограничения;
- файлы и входные данные.

Так модель точнее, чем при коротких общих запросах.

## Основные техники управления контекстом

### History compression

История диалога сжимается в summary, идёт дальше как новая база контекста.

Плюс: продолжаешь диалог после переполнения окна.

Минус: теряются детали, появляются ошибки и галлюцинации. Для сложных задач, особенно с кодом, нежелательно, но иногда неизбежно.

### Sliding Window

Окно сдвигается: старые сообщения удаляются, свежие остаются.

Хорошо для одной темы. Плохо если юзер пишет в один чат много несвязанных задач.

### Vector embeddings

Данные → векторы, хранятся внешне, подгружаются по релевантности.

Не держишь весь текст в контексте, достаёшь только нужные фрагменты. Связано с RAG, экономит окно.

### Context layering

Контекст по слоям:

- стратегический слой проекта;
- слой текущей фичи;
- слой конкретной задачи.

Агент получает только нужную часть, не всю историю проекта.

### Пересоздание диалога

Для новой задачи лучше очистить контекст, начать новый диалог. Меньше шума, агент получает «чистый лист».

### Токенизация и обрезка

Удаляй малозначимые токены: вежливые фразы, повторы, технический шум, лишние скобки, неважные уточнения.

Экономит окно, но не бесконечно: при длинной работе всё равно сжатие или разбиение задач.

### Topic branches

Система дробит один длинный чат на тематические ветки. Юзеру диалог выглядит единым, внутри контекст разделён по темам.

## Контекстное окно

Большие модели работают с очень крупным контекстом, вплоть до миллионов токенов. Но контекст не бесконечен.

Для кода окно расходуется быстро:

- одно взаимодействие с код-задачей ≈ 8 000–10 000 токенов;
- 1 миллион токенов ≈ 100 таких сообщений;
- длинная разработка быстро требует чистить, дробить, сжимать контекст.

Поэтому для код-агентов проектируй управление контекстом заранее.

## Управление памятью

### Типы памяти

1. **Без памяти** — модель отвечает на текущий запрос, ничего не сохраняет.
2. **Session memory** — краткосрочная память в пределах диалога.
3. **Long-term memory** — долговременная, агент узнаёт юзера, подгружает прошлые вводные.

### Компрессированная память

Диалоги или этапы сжимаются в summary и сохраняются.

Плюсы:

- мало токенов;
- хранишь длинную историю;
- удобно подмешивать в системный промпт.

Минусы:

- теряются детали;
- summary недетерминировано;
- важные нюансы исчезают;
- риск галлюцинаций растёт.

### Vector-based memory

Факты и фрагменты диалога сохраняются как embeddings в векторной базе.

Плюсы:

- хорошо масштабируется;
- хранишь много деталей;
- удобно доставать релевантные фрагменты.

Минусы:

- нужны доп. модели и вычисления;
- retrieval не гарантирует правильные выводы;
- сохранённые факты ≠ корректное рассуждение.

### Слоистая память

Память по уровням и приоритетам: общий контекст, контекст направлений, детали под задачи.

Плюсы: точность и масштабируемость.

Минус: сложно спроектировать слои.

### Rule-based memory

Память на правилах вроде «если суши, вероятно японская кухня».

Плюсы:

- контроль;
- детерминированность;
- предсказуемое поведение в рамках правил.

Минусы:

- негибкость;
- дорогая поддержка;
- всё за пределами правил трактуется хуже.

### Self-reflective memory

Сохраняются не только факты, но и семантические смыслы:

- стиль общения юзера;
- предпочтения;
- настроение;
- темы интереса;
- реакции на форматы.

Создаёт эффект «узнавания», повышает персонализацию. Но такую память фильтруй, обновляй, поддерживай — смыслы меняются.

## Комбинации памяти

Современные ассистенты используют комбинации, не один подход:

- простой ассистент: текущий контекст и ответ из общей базы знаний;
- умный ассистент: сессионная память, векторы, правила;
- продвинутый ассистент: иерархическая память, self-reflection, vector retrieval, смысловые слои.

Чем лучше скомбинировано, тем сильнее ощущение, что ассистент «помнит» юзера и понимает контекст.

## Планирование

Планирование особенно нужно для своих агентов и код-ассистентов.

Базовый процесс:

1. Определить цель.
2. Разбить на задачи.
3. Выбрать стратегию исполнения.
4. Перепланировать, если стратегия не подходит.
5. Подтвердить план у юзера.
6. Выполнить задачи.

Похоже на Planning Mode в код-ассистентах: агент формирует план, юзер подтверждает, потом реализация.

## Tool orchestration

Инструменты дают агенту связь с внешним миром: API, CLI, MCP-серверы, bash-команды, файловая система, другие интерфейсы.

Этапы работы:

1. выбрать инструмент;
2. вызвать;
3. получить результат;
4. интерпретировать;
5. обработать ошибки;
6. обновить состояние задачи.

Критична интерпретация результата. Инструмент может вернуть технически корректный ответ, но агент должен понять, решает ли он исходную задачу.

## Практические лайфхаки

### 1. Сужать контекст

В каждом серьёзном запросе указывай язык, стек, задачу, файлы, ограничения, ожидаемый формат. Меньше неопределённости — лучше работа агента.

### 2. Индексировать и векторизовать проект

Для кодовых проектов индексируй файлы, строй связи, подгружай агенту только релевантные части.

### 3. Документировать проект

Комментарии, сигнатуры функций, краткие описания модулей дают агенту понять код без всех исходников. Экономит токены.

### 4. Работать по стадиям

Большую задачу бей на этапы, передавай каждый этап новому агенту с сохранённым состоянием. Каждый этап получает своё окно.

### 5. Избегать частой суммаризации

Суммаризация теряет детали. Для долгой работы с кодом лучше дроби задачи, используй индексацию, слои контекста, сохранение состояния.

## Промпты

### Плохой короткий промпт

> Напиши функцию на Kotlin, которая проверяет, является ли число простым.

Проблема: мало контекста. Неясно где используется код, какой стек, стиль, ограничения, тесты, среда.

### Pro-промпт

Лучше укажи: модуль для Android-приложения, конкретный стек, протокол, Google-аутентификация, документация, ограничения. Тогда модель понимает задачу точнее.

### Prompt Pro Max

Ещё лучше — мета-промптинг: попроси нейросеть создать подробный промпт со стеком, примерами, ограничениями, форматом результата.

Сохрани такой промпт как slash-команду. Занимает больше токенов, но часто экономит больше за счёт меньшего числа уточнений и ошибок.

## JSON и TOON

JSON — индустриальный стандарт, но расходует много токенов, особенно из-за повтора имён полей в массивах объектов.

Для своих агентов с оплатой per token дорого.

TOON (Token Oriented Object Notation) — компактнее:

- структура полей задаётся один раз;
- значения компактнее;
- повтор ключей сокращается;
- расход токенов меньше.

Экономия особенно заметна на массивах объектов с одинаковыми полями.

## Цель недели

Итог недели 2:

1. разобраться с контекст-менеджментом;
2. разобраться с памятью агентов;
3. построить простой flow агента;
4. научиться управлять контекстом, состоянием, обработкой данных;
5. подготовиться к теме MCP и инструментов.

Главный вывод: хороший агент — не только модель и промпт. Это система, где правильно организованы контекст, память, планирование, инструменты, стоимость токенов, разбиение задач на стадии.
