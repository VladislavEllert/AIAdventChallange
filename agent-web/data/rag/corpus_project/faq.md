# FAQ поддержки — частые проблемы и решения

Собрано из реальных тикетов поддержки (`agent-web/data/support/tickets.json`) и
исторических багов проекта (`memory-bank/progress.md`, недели 6-7).

## Локальные модели / прокси на Windows

**Симптом:** запросы к Ollama/ComfyUI/MCP/`/api/metrics` виснут или падают с ошибкой
соединения через прокси, хотя сервер поднят и слушает нужный порт.

**Причина:** httpx и OpenAI SDK по умолчанию доверяют системным env-переменным прокси
(`trust_env=True`). На Windows системный SOCKS-прокси перехватывает даже локальные и
LAN-адреса (localhost, `192.168.x.x`).

**Фикс:** явно передавать `trust_env=False` при создании HTTP-клиента для любого
local/LAN/VPS-вызова (Ollama, ComfyUI, metrics-прокси, MCP-серверы). Для облачных
вызовов через ProxyAPI прокси может быть нужен — там `trust_env` не трогаем.

## Ollama qwen3 возвращает пустой ответ

**Симптом:** RAG-запрос на `ollama/qwen3:4b` возвращает пустую строку; обычный чат без
RAG работает нормально.

**Причина:** qwen3 — thinking-модель, генерирует скрытый блок `<think>...</think>` перед
финальным ответом. Длинный RAG-контекст в промпте увеличивает расход токенов на
размышление, и при `max_tokens` порядка 1500 модель успевает потратить весь лимит на
`<think>`, не дойдя до видимого ответа. `think:false` и `/no_think` не сработали на
использованной сборке Ollama.

**Фикс:** для `ollama/*` моделей query-rewrite (перевод запроса на английский, день 23)
отключён — это укорачивает промпт и снижает риск исчерпания лимита. При необходимости
дополнительно поднять `max_tokens` для ollama-моделей отдельно от облачных.

## sessions.db повреждена после жёсткого убийства процесса

**Симптом:** после `taskkill /F` (Windows) или `kill -9` поверх открытой WAL-транзакции
SQLite-файл `sessions.db` повреждён, приложение падает на старте.

**Фикс:** останавливать процесс через Ctrl+C / SIGINT (штатное завершение), не `/F`/`-9`.
При повреждении — пересоздать `sessions.db`, важные данные там не хранятся (только
активные сессии чата).

## Мобильная вёрстка (узкий viewport)

**Симптом:** на телефоне (< 480px) боковые панели (Settings/Memory/Task) перекрывают
чат без возможности скрыть, тап-зоны меньше 44px, фокус на поле ввода триггерит iOS
Safari zoom.

**Фикс:** панели переведены на выезжающие drawer'ы с overlay, минимальный размер
тап-таргетов поднят до 44px, `font-size` инпута зафиксирован от 16px (убирает
автозум iOS Safari при фокусе).

## `/api/metrics` виснет на несколько секунд

**Симптом:** запрос к `/api/metrics` (проксирует метрики с Windows-бокса по LAN) висит
секунды даже когда remote-хост недоступен.

**Причина:** недоступность LAN-хоста — не "instant refused" как для localhost, а полный
сетевой таймаут. Дефолтный таймаут httpx-клиента был выставлен слишком высоко (3.0с).

**Фикс:** таймаут прокси-клиента снижен до 0.8с — HUD в статус-баре деградирует в
"offline" быстро, не блокируя основной UI.

## `.env` резолвится не в тот файл

**Симптом:** переменная окружения (например `OLLAMA_CHAT_URL`) из `agent-web/.env`
игнорируется, используется дефолт, хотя файл существует и заполнен верно.

**Причина:** `python-dotenv` по умолчанию ищет `.env` через frame-based поиск вверх по
стеку вызовов — при наличии одновременно root `.env` и `agent-web/.env` резолвится не
тот файл в зависимости от того, откуда запущен процесс.

**Фикс:** путь к `.env` передавать явно (`load_dotenv(explicit_path)`), не полагаться на
автопоиск.

## Стриминг ломается под DispatchProvider

**Симптом:** после введения `DispatchProvider` (роутинг `ollama/*`/`comfyui/*`/остальное
по префиксу модели) стриминг облачных ответов пустой или падает.

**Причина:** проверка `isinstance(provider, ProxyAPIProvider)` в
`respond_stream_with_stats` не проходит для `DispatchProvider`, даже когда тот
маршрутизирует именно на ProxyAPI внутри себя.

**Фикс:** проверка типа заменена на духless-проверку возможностей провайдера, не на
конкретный класс.

## DispatchProvider требует ProxyAPI-ключ даже для локальных моделей

**Симптом:** приложение падает на отсутствующем `PROXYAPI_KEY`, хотя используются только
`ollama/*`/`comfyui/*` модели.

**Причина:** `ProxyAPIProvider` внутри `DispatchProvider` создавался эагерно в
конструкторе, независимо от реально используемой модели.

**Фикс:** ленивая конструкция — `ProxyAPIProvider` создаётся только при первом реальном
обращении к облачной модели.
