## Задача

Реализовать в агенте **3 стратегии управления контекстом** + переключатель, прогнать один
сценарий («сбор ТЗ, 10–15 сообщений») на каждой и сравнить качество, стабильность, расход
токенов, удобство:
1. **Sliding Window** — хранить только последние N сообщений, остальное отбрасывать.
2. **Sticky Facts / KV-память** — блок ключ-значение, обновляемый после каждого сообщения юзера, + последние N.
3. **Branching** — checkpoint, ветки от одной точки, независимое продолжение, переключение.

## Архитектура решения

- **Не тумблеры-настройки, а 3 именованных агента.** Каждый привязан к своей стратегии.
  «Переключатель» из задания = выбор агента. Так наглядно для демо и не плодит ручных флагов.
- **DRY/KISS:** один протокол `ContextStrategy`, 3 реализации. Агент делегирует сборку контекста
  стратегии; стандартный путь «моих» агентов (summary-стек дня-9) не тронут.
- **Приложение поделено надвое** верхним сегмент-тумблером **«Мои агенты | Тестовые»**
  ([HomeView.swift](../../AgentChat/AgentChat/Views/HomeView.swift)): «Мои» — best-practice агенты
  (summary), «Тестовые» — курсовой полигон с этими 3 стратегиями (без summary). Основной интерфейс
  не захламляется экспериментами.

### Стратегии

- **🪟 Окно N (Sliding Window).** В модель уходят только последние N сообщений; старое за окно
  не передаётся. Токены не растут — но всё, что вышло за окно, агент забывает.
- **🗂️ Факты KV (Sticky Facts).** После каждого сообщения юзера дешёвая модель извлекает
  KV-блок (Цель / Ограничения / Предпочтения / Решения / Договорённости). В запрос идёт
  `KV-блок + последние N`. Ключевое держится в фактах, даже если вышло за окно. Блок помечен
  как служебная память — агент не печатает его списком.
- **🌿 Ветки (Branching).** Ручной форк от любого сообщения (checkpoint). Ветка = независимая
  линейная нить (копия префикса + свои сообщения). Чипсы-переключатель, ⭐ основная ветка,
  удаление/назначение основной. Активная ветка шлётся целиком, без сжатия. Ветки изолированы —
  контекст одной не течёт в другую.

## Как проверить

Раздел **«Тестовые»** → 3 агента. Рабочая модель — `gpt-4o-mini`.

1. **Окно N:** дай факт в 1-м сообщении, веди 8+ реплик про другое, спроси факт в конце →
   **забыл** (вышел за окно). Токены в HUD не растут.
2. **Факты KV:** тот же сценарий → деталь **помнит** (она в KV-блоке), даже за окном. Видна
   лента фактов + счётчик.
3. **Ветки:** база 2 сообщения → форк → «Ветка 1» (стек Swift) → назад в «Основную» →
   ещё форк → «Ветка 2» (дизайн) → кросс-проверка: каждая ветка знает только своё.

## Выводы по стратегиям

**🪟 Sliding Window.** Очень хорошо экономит токены, но для памяти неэффективен — просто
забывает то, что вышло за окно. Дал бы его под **обычные задачи**, где память не важна:
хочешь поговорить с нейронкой и тебе неважно, что сохранится в истории, и не хочешь тратить
много токенов — этот способ подходит. Огромный плюс — сильная экономия. Дальше всё зависит
от задачи.

**🗂️ Sticky Facts (KV).** Очень удобно планировать в такой стратегии. Качество на высоте,
**очень стабилен — вообще не теряет важные детали**. Расход токенов не сказал бы что большой,
и безумно удобно. Минус: со временем расход будет расти — фактов становится много.

**🌿 Branching.** Прикольно, что можно вести **изолированные диалоги, чтобы контекст не мешался**.
Классная фича для большого проекта: хочется без подмеса контекста обсудить ту или иную фичу
отдельно, не тратя сильно токены — это круто. Это **не про экономию токенов, а про качественные
и стабильные ответы строго по теме**. Можно форкнуть одно сообщение и разбить на разные линии.
В идеале работает так: поговорили в разных ветках → из каждой взяли решение → слили в основную
ветку готовый стек. Безумно удобно.
