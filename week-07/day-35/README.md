## Какую реальную задачу это автоматизирует

Каждый день этого челленджа заканчивается одним и тем же ручным действием: открыть
`README.md` (корень) и `memory-bank/progress.md`, найти таблицу прогресса, дописать
строку с тем, что сделано за день — статус, ссылка на код, ссылка на видео (или `todo`).
За 34 дня это действие сделано вручную минимум 34 раза, и оно легко ломается тремя
типовыми способами:

1. **Забыть один из двух файлов** — README и progress.md дублируют одну и ту же таблицу;
   обновить один и забыть другой — обычная рассинхронизация вручную.
2. **Сломать формат таблицы** — markdown-таблица без валидатора: пропущенный `|`,
   не тот порядок колонок, лишний пробел — молча ломает рендер на GitHub/GitVerse.
3. **Выдумать ссылку на видео** — самое опасное: скопировать шаблон прошлого дня и
   забыть заменить `todo` на реальную ссылку (или наоборот, оставить старую ссылку не
   с того дня). CLAUDE.md прямо запрещает придумывать несуществующие ссылки на видео —
   это тот случай, где автоматизация может нарушить правило быстрее и незаметнее, чем
   человек за клавиатурой.

`/ritual day <NN>` — это цепочка из трёх простых функций (без фреймворка субагентов,
без параллелизма — специально: fan-out здесь ничего не выигрывает, все три шага
последовательно зависят друг от друга), которая делает эту рутину: собирает реальный
`git diff`/`git status`/метрики ревью (без LLM — детерминированный сбор фактов),
просит LLM (`openai/gpt-4o-mini`, единственный живой вызов в цепочке) составить строку
таблицы **строго из собранных фактов**, затем детерминированно (без LLM — та же логика,
что и `danger.py` дня 34: это не творческая задача, а проверка формата) отклоняет
черновик, если формат таблицы сломан, ссылки нет, или строка утверждает наличие
видео-ссылки, которой заведомо нет в собранных фактах. Только после этого — реальный
diff в чат, подтверждение человеком через ТУ ЖЕ модалку, что и день 34, и **только
после Allow** — реальная запись на диск + `git commit` двух файлов. Push никогда не
делается — ни автоматически, ни за флагом: `git_push`-тула в кодовой базе нет вообще.

## Что было сделано

### 35.1-35.3 — цепочка + tools + команда

`day_report.py`: `collect()` читает `git status --porcelain`, сортирует изменённые файлы
untracked/added-first (см. живой баг ниже — почему это важно), берёт по каждому файлу
свой бюджет символов на diff (не единый `diff[:N]` на весь репо), плюс агрегат из
`review_metrics.jsonl` если он есть. `draft()` — единственный LLM-вызов, system-промпт
явно требует `Видео = todo`, если фактов о видео нет, и требует ссылку в формате
`[week-WW/day-DD](week-WW/day-DD/)`. `verify()` — чистый Python: 6 колонок, `Статус`
∈ {done, todo}, markdown-ссылка в `Код`, и **любая** непустая claim в `Видео` кроме
буквального `todo` отклоняется как галлюцинация (у пайплайна физически нет источника
правды про видео — значит любое утверждение о нём недоказуемо по построению).
`build_patch()` — upsert строки по номеру дня (замена, если день уже есть в таблице,
вставка после последней `| 07 | NN |` строки иначе), unified diff для показа.

`git_tools.py`: `git_diff`/`git_log`/`git_status` — safe, но **намеренно не
зарегистрированы** в общий `services/tools/registry.py` (в отличие от `fs_tools.py`).
Причина: `mcp-server/project_server.py` (день 31) уже отдаёт MCP-тулы с ИДЕНТИЧНЫМИ
именами (`git_status`/`git_diff`). `chat.py` мёржит `mcp_schemas + local_schemas` в один
список для LLM при `use_mcp=True` — OpenAI API **отклоняет** `tools`-массив с двумя
функциями одного имени (400). Раз `services/tools/registry` — процесс-широкий singleton
dict, регистрация тут задела бы КАЖДЫЙ существующий чат с `use_mcp=True`, а не только
`/ritual` — реальный риск регресса дней 31/33/34. `git_commit` коллизии не имеет (нет
MCP-аналога) — зарегистрирован как обычно, danger_level=`dangerous` в `danger.py`.

`commands_ritual.py` реиспользует ВЕСЬ confirm/executor day 34 напрямую (тот же
`tools_executor.execute_stream()`, тот же `ConfirmToolModal.tsx` во фронтенде — новых
UI-компонентов день 35 не добавляет). Потребовалась одна маленькая правка `chat.py`:
диспетчер слэш-команд теперь передаёт `stream_id` в `handler(...)` (был нужен
`git_commit`'у для confirm-хендшейка) — `handle_help`/`handle_support` получили
неиспользуемый `stream_id: str = ""` для совместимости сигнатуры, поведение не менялось.

### Живой баг, пойманный при первом же прогоне (до коммита)

Первый `--dry-run` прогон против реального диффа недели (репозиторий не коммитился
с дней 31-34, ~85 изменённых файлов, `git diff` 645KB) вернул строку про **день 32**
(AI-ревью PR), а не про день 35. Причина: наивный `diff[:6000]` брал первые 6000 символов
единого `git diff`, который git печатает в алфавитном порядке файлов — `AGENTS.md`,
`README.md`, `agent-cli/agent_cli/config.py`... ни один файл дня 35 (`rituals/`,
`git_tools.py`, `commands_ritual.py`) физически не попадал в окно. Исправлено:
`collect()` сортирует untracked/added файлы (`??`/`A` в `git status --porcelain`) ПЕРЕД
изменёнными — новые файлы это самый сильный сигнал "это сделано именно сегодня" — и
строит diff, выдавая каждому файлу свой бюджет символов в этом порядке, вместо слепого
среза одного большого текста.

После фикса второй прогон улучшился, но не идеально: модель дважды подряд включила в
ответ ЗАГОЛОВОК таблицы (`| Неделя | День | ... |` + `|---|---|...|`) перед самой
строкой данных, несмотря на прямой запрет в промпте — верификатор корректно это поймал
(`expected 6 columns... got 20`) и отклонил черновик, как и задумано. Добавлена defensive
пост-обработка `_extract_data_row()` в `draft()`: берёт последнюю непустую строку,
которая не заголовок и не разделитель — no-op, если модель и так выдала одну строку;
покрыто регресс-тестом (`test_draft_strips_header_and_separator_if_model_echoes_them`).

Третья итерация (после усиления промпта: "based STRICTLY on NEW FILE entries") дала
приемлемый, но не идеальный результат — модель всё ещё иногда перечисляет `/help` и
`/support` как будто они добавлены сегодня (это команды дней 31/33, не 35). Ни один из
фактов не выдуман (ссылка реальная, статус реальный, видео = `todo`) — это неточность
формулировки, не галлюцинация в смысле CLAUDE.md. Дальше промпт-инжиниринг не гонялся —
не цель фазы; финальная запись в README/progress.md для дня 35 сделана вручную
(тем же `day_report.py`, headless CLI, но с ревью диффа человеком перед `y`) — см.
раздел "живой прогон" ниже.

### 35.6 — PROMPT_V2_STRICT

`SYSTEM_PROMPT_V2_STRICT` добавляет явный чеклист из 5 реально повторявшихся в этом
проекте классов багов (dotenv не тот файл, `isinstance` ломается под обёрткой,
`httpx`/`trust_env` подхватывает системный прокси, широкий `except:`, путь без
sandbox-проверки) и просит короткое обоснование "какой именно класс" на каждый найденный
баг. Прогон в одну сессию, тот же RAG-индекс (`4f7971c234d9`), только промпт разный:

| prompt_version | recall | FP-rate | latency p50 | cost/прогон |
|---|---|---|---|---|
| `v1` | 33% (1/3) | 0% (0/1) | 2815ms | 0.0385₽ |
| `v2_strict` | **100% (3/3)** | 0% (0/1) | 2688ms | 0.0538₽ |

`v2_strict` поймал фикстуру 03 (httpx `trust_env`) — единственную, которую оба
предыдущих `v1`-прогона (день 32 и день 33) пропускали. Не переключено на прод
(`PROMPT_VERSION` в CI остаётся `v1`) — 4 фикстуры мало для claim "обобщается", это
демонстрация механизма сравнения, не финальное решение. Полные данные:
[`review_eval/results_day32.md`](../../agent-web/review_eval/results_day32.md), Прогоны 3-4.

### 35.4 — живой прогон

`--dry-run` прогонялся многократно против реального диффа недели (см. баги выше). Ещё
один живой баг черновик LLM тоже не устранил полностью — после третьей итерации промпта
модель всё ещё иногда приписывала `/help`/`/support` (дни 31/33) к сегодняшней работе,
хоть и без выдумывания фактов (ссылка/статус/`todo`-видео — всё реальное). Дальше
промпт-инжиниринг не гонялся — не цель фазы. Финальная строка для README/progress.md
написана вручную (человеком-агентом) и прогнана через ТУ ЖЕ `verify()`/`build_patch()`,
что и живой ритуал — только `chat_fn` заменён на стаб, возвращающий проверенный текст,
вместо доверия сырому выводу модели. Это и есть "review the proposed diff yourself
before approving" буквально.

При реальном коммите (`git_tools._git_commit`) найден ещё один живой баг: `git commit`
без pathspec коммитит ВЕСЬ индекс, а не только то, что только что застейджено этим же
вызовом. `week-07/day-34/TOOLS_DOC_DRAFT.md` был застейджен ЗАРАНЕЕ (день 34 готовил
живой git-статус-бейзлайн для своего e2e) и незаметно попал в day-35 коммит вместе с
README/progress.md — обещание "стейджит ТОЛЬКО переданные пути" держалось для `git add`,
но не для `git commit`. Исправлено: `git commit -m msg -- <paths>` — pathspec на самом
коммите, не только на add, независимо от того, что ещё лежит в индексе. Регресс-тест
`test_git_commit_never_sweeps_in_other_already_staged_files` добавлен. Побочный эффект:
`TOOLS_DOC_DRAFT.md` теперь закоммичен (был просто застейджен) — обновлён бейзлайн в
`tool-confirm.spec.ts` (`BASELINE_STATUS` `"A  <path>"` → `''`, позитивный сценарий
`"AM"` → `"M"`, чистый tracked-файл вместо staged-added), день-34 сценарии перепрогнаны
и снова зелёные.

Коммит для дня 35 (README.md + memory-bank/progress.md) — локальный, push не
выполнялся. `grep -rn "git push" agent-web/agent_web/services/tools/ agent-web/agent_web/services/rituals/ agent-web/agent_web/services/commands_ritual.py` — ничего, кроме упоминания в докстринге о его отсутствии.

### E2E / pytest

`pytest agent-web/tests` — все тесты зелёные (счёт в отчёте агента). Playwright
`ritual-command.spec.ts`: dry-run (черновик + diff в чате, без модалки) и
confirm+Deny (модалка появляется, репо не тронут после отказа) — оба сценария живые,
реальный `openai/gpt-4o-mini`. Позитивный Allow-сценарий **намеренно не автоматизирован
в Playwright** — см. докстринг спека: коммит README/progress.md (живых, "боевых"
документов проекта, не одноразового черновика как `TOOLS_DOC_DRAFT.md` дня 34) на КАЖДЫЙ
повторный прогон CI — плохой паттерн для файлов, которые редактируются вручную и должны
проходить ревью один раз, а не при каждом re-run тестов.
