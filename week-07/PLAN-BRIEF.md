## 5 слоёв (майлстоуны плана)

### День 31 — Ассистент разработчика (фундамент)
- RAG-индекс по докам проекта: README, `agent-web` docs, схемы данных/API.
- **+ Индексировать конспекты лекций** `memory-bank/lessons/` — агент становится «гуру по курсу»,
  знает принципы Гладкова и что мы проходили.
- MCP-тул минимум: текущая git-ветка (`git rev-parse --abbrev-ref HEAD`), опционально diff/список
  файлов.
- `/help` — отвечает на вопросы о проекте через RAG + контекст. Пример, который ОБЯЗАН тянуть:
  «как реализован RAG?» → «Ollama embeddings + JSON-индекс…» из собственного проиндексированного кода.

### День 32 — Автоматизация ревью кода
- GitHub Action на PR: берёт diff + изменённые файлы, гонит через тот же RAG (доки+код), возвращает
  ревью (потенциальные баги / архитектурные проблемы / рекомендации).
- Логика ревью — headless python-модуль, вызываемый и Action'ом, и вручную (один код, два входа).
- Нужны синтетические тестовые PR в agent-web (придумать правки для теста пайплайна).
- Лекция акцентирует (см. §Лекция): error handling / retry / fallback / human-in-the-loop /
  метрики (latency p50/p95/p99, cost, quality-гейт), тест на эталонных старых PR + smoke test.

### День 33 — Ассистент поддержки пользователей
- Саппорт-ассистент поверх той же RAG-базы + MCP к синтетическому `tickets.json`.
- Тикеты строим из **реальных багов из `memory-bank/progress.md`** (напр. «почему не работает
  авторизация»). Учитывает контекст тикета. Вариант «JSON тикетов», не реальная CRM.

### День 34 — Ассистент для работы с файлами
- Читает/ищет по нескольким файлам, анализирует, создаёт/меняет файлы. Мин. 2 сценария из списка
  задания. Агент САМ инициирует работу с файлами (задача на уровне цели, не «открой файл X»).
- Изменения сохраняются или выводятся как diff, воспроизводимо.
- **Опасные операции (удаление/commit/push) — human-in-the-loop подтверждение** (лекция это прямо
  требует, по образцу инвариантов).

### День 35 — Реальная задача
- Агент дня 34 закрывает НАШ ЖЕ ритуал курса: видит новый день/diff → актуализирует README +
  `progress.md` → коммитит. Замыкает неделю (последний день использует выращенного агента).
- Идея инвест-бота ОТКЛОНЕНА для дня 35 (новый стек, риск не уложиться за день).

## Требования к качеству билда (от юзера)

- Пошагово, самодостаточно, всё покрыть тестами (pytest).
- Агент должен САМ тестировать через браузер (Playwright). Если не умеет — **написать скилл**,
  чтоб умел сам протыкивать golden path (см. memory `feedback-drive-golden-path`).
- «Готово» рапортовать ТОЛЬКО после реального теста (код + клик в браузере).
- Агенту дать полный доступ к терминалу (запускает что нужно).
- Не ограничены по времени/токенам, планка качества высокая, минимум участия юзера.

## Что учесть из лекции недели 7 (транскрипт)

Файл: `memory-bank/lessons/` (загрузить конспект потока 6). Ключевое:
- LLM-агент = мини-ОС; субагенты (fan-out / chain / router паттерны) для сложных недетерминир. задач.
- System tooling: Rack tools + MCP tools + System tools (bash/файлы). Tool registry + executor.
- **Опасные операции → спрашивать человека** (git push, удаление, конфиги).
- Pipeline: trigger (PR/расписание/событие) → input → agent(LLM+RAG+tools) → output. День 32 —
  прямой пример из лекции.
- Error handling обязателен: retry (др. промпт/модель), fallback-стратегия, human-in-the-loop,
  никогда не полностью автоматом.
- Метрики: latency (p50/p95/p99), cost на прогон, quality-гейт (оценка человеком), % успешных прогонов.
- Тест пайплайна: эталонные старые PR, сравнение версий промптов, smoke test, оценка человеком.
- Дух недели: не Hello World — реальный продукт, рефакторинг/архитектура, красиво.
