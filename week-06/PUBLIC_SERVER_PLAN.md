## Контекст

`agent-web` (FastAPI + React) уже развёрнут и работает на Windows-машине (RTX 4060, 8GB):
локальный чат (`ollama/qwen3:4b`), генерация картинок (ComfyUI/SDXL), RAG, MCP, метрики — всё
живьём проверено, LAN-доступ работает (`http://192.168.0.33:8765`, порт 8765 открыт в firewall).

**Чего не хватает:** публичного HTTPS-доступа. Юзер хочет зайти на свой сайт с телефона **не по
домашнему WiFi** (через мобильный интернет) и дать ссылку доверенным друзьям/сокурсникам.

**Что уже пробовали и почему не вышло:** Cloudflare Tunnel (`cloudflared`) — и QUIC, и HTTP/2 до
`argotunnel.com` не проходят (TLS handshake рвётся с EOF на HTTP/2, QUIC — timeout). Похоже на
целевую блокировку протокола (не общий сетевой сбой — обычный HTTPS к другим хостам работает
нормально). Прокси (SOCKS через VPN-софт на машине) не помогает — QUIC не идёт через
env-переменные прокси, HTTP/2-режим cloudflared тоже не пробился. Подробности и точные логи
ошибок — в `week-06/WINDOWS_SETUP.md` §ШАГ 11 (последний абзац, «Публичный доступ отложен»).

**Важная деталь среды:** на машине настроен системный SOCKS-прокси (VPN-софт, обязателен для
работы машины — НЕ отключать). Он ломает голые `httpx`/`OpenAI`-клиенты без `trust_env=False` —
если правишь Python-код, который делает сетевые запросы, помни про это (уже пофикшено в
`agent_cli/llm/ollama.py`, `agent_web/services/comfyui_client.py`, `agent_web/routers/metrics.py`,
`agent_web/services/mcp_client.py` — паттерн одинаковый, ищи `trust_env=False` в этих файлах для
примера).

## Цель

Публичный HTTPS-URL, с которого можно достучаться до `agent-web` на этой Windows-машине снаружи
домашней сети — с телефона по мобильному интернету, с любого компьютера. Цель — «до талого»:
пробовать варианты по очереди, пока один не заработает по-настоящему (не «вроде должно», а
проверено `curl` с внешней сети или хотя бы с телефона с выключенным WiFi).

## Кандидаты (пробовать по порядку, каждый — с проверкой + чисткой при провале)

### 1. Диагностика: за NAT ли машина, есть ли CGNAT
Прежде чем пробовать port-forward — узнать, вообще ли это возможно.
```powershell
curl.exe -s https://api.ipify.org
ipconfig | findstr IPv4
```
Сравнить внешний IP (ipify) с IP на роутере (если есть доступ к админке роутера, обычно
`192.168.0.1` или `192.168.1.1` — смотреть WAN IP там). Если они совпадают — прямой белый IP,
port-forward может сработать. Если разные — провайдер, скорее всего, за CGNAT, и port-forward
с роутера не поможет вообще (нужен VPN-провайдер с публичным IP или туннель-сервис).

### 2. ngrok (быстрый, отдельная инфраструктура от Cloudflare)
```powershell
winget install ngrok.ngrok
ngrok config add-authtoken <токен с ngrok.com, юзер зарегистрируется сам, это бесплатно>
ngrok http 8765
```
Проверить выданный `https://*.ngrok-free.app` URL curl'ом снаружи (или попросить юзера открыть с
телефона на мобильном интернете). Если тоже блокируется на сетевом уровне — это сильный сигнал,
что блокировка не специфична для Cloudflare, а бьёт по всем туннель-сервисам разом; в отчёте
отметь это явно, не трать время на похожие туннель-сервисы (localtunnel и т.п. используют
похожую архитектуру, вероятность успеха низкая, но если время останется — можно попробовать
`localtunnel` (npm, `npx localtunnel --port 8765`) как быстрый доп. тест этой гипотезы.

### 3. Tailscale Funnel
Отдельная архитектура (WireGuard-based mesh VPN + Funnel = публичный HTTPS-выход), не пробовали
ещё. Может обходить блокировку, т.к. транспорт другой (не QUIC/HTTP2 к argotunnel).
```powershell
winget install Tailscale.Tailscale
tailscale up
tailscale funnel 8765
```
Проверить выданный URL (`https://<machine>.<tailnet>.ts.net`) снаружи.

### 4. VPN-провайдер с проброшенным портом
Если у юзера на этой машине уже стоит VPN-клиент (тот самый, что даёт SOCKS-прокси) — узнать у
юзера, какой это провайдер, и есть ли у него функция port-forwarding (некоторые VPN дают
статический внешний IP + проброс порта в личном кабинете — например, у части провайдеров это
платная опция). Если есть — пробросить 8765 через VPN-провайдера, не через домашний роутер.

### 5. Router port-forward + Dynamic DNS (если п.1 показал белый IP)
Если внешний IP реально белый (не CGNAT):
- Зайти в админку роутера (спросить юзера логин/пароль или сделать вместе), пробросить порт 8765
  (или 443, если получится поставить TLS) на `192.168.0.33:8765`.
- DDNS (например, `noip.com` или `duckdns.org`, бесплатные) — чтобы не зависеть от смены IP.
- HTTPS: без домена+сертификата будет только HTTP по IP — для нормального HTTPS нужен домен
  (можно бесплатный поддомен DDNS) + Let's Encrypt (`certbot`) или self-signed (браузер будет
  ругаться, но с телефона можно принять исключение).
- Это самый «настоящий» вариант, но требует доступа к роутеру и может не сработать, если
  провайдер режет входящие 80/443 (частая практика для домашних тарифов) — если так, пробовать
  нестандартный порт (например 8765 напрямую, без 443).

## Что проверить в конце (обязательно)

- Финальный рабочий URL открывается **с внешней сети** (телефон, WiFi выключен, только мобильный
  интернет) — не просто "curl ответил", а реально видно UI, можно отправить сообщение в чат и
  получить ответ.
- Все неудачные попытки убраны: `winget list` / `Get-Process` / `netstat -ano` — ничего лишнего
  не висит в фоне; лишние firewall-правила (`Get-NetFirewallRule -DisplayName "*ngrok*"` и т.п.)
  удалены, если сервис не пригодился.
- `git status` в ветке чист (или все изменения осмысленно закоммичены).
- Итоговый отчёт юзеру: что сработало (или честно — что не сработало и почему), финальный URL
  (если есть), что теперь нужно для поддержания (например, ngrok бесплатный tier даёт новый URL
  при каждом перезапуске — если так, сказать явно).
- Запушить ветку (`git push origin feature/public-https-server`) в конце, НЕ мержить в `main`.
